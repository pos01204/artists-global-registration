## idus Global 작가 온보딩 페이지 리뉴얼 개선안 (Design + IA + Backend/Railway)

### 0) 문서 목적
- **목표**: 현재 온보딩 페이지가 “깔끔하지만 템플릿/AI 느낌”으로 인지되는 문제를 해소하고, **idus Global의 브랜드 톤**(신뢰/친근/프로페셔널)을 강화하면서도 **전환(학습→등록)**에 최적화된 UI/UX로 리뉴얼하는 실행 가능한 개선안을 정리합니다.
- **범위**: 프론트(정보구조/디자인 시스템/컴포넌트/카피) + 데이터 수집/아카이빙(구글시트) + **Railway 백엔드 배포 아키텍처**.
- **기본 원칙(유지)**: 참여 피로도 최소화(페이지당 카드 3개 이하), 정보 누락 없이, “작가님이 할 일은 적다”를 명확히.
- **이번 보완 포인트(추가 요청 반영)**:
  - **Pretendard 폰트 적용 + 굵기(Weight) 위계 정교화**: 요소 역할에 따라 Bold 강도를 세분화
  - **Google Sheets는 “아카이빙(로그)”만**: 검증/중복 방지/리트라이/집계 등 처리 로직은 **Railway로 이관**

### 1) 현행 구조 요약(현재 코드 기준)

#### 1-1. 라우팅/흐름
- **`/`**: 작가 정보 입력(이름/연락처/사업자/카테고리)
- **`/qualification`**: 준비 완료(학습 CTA)
- **`/learn`**: 학습 목차(스텝 진입)
- **`/learn/step/[stepId]`**: 스텝별 콘텐츠(페이지형 슬라이드)
- **`/learn/quiz`**: 퀴즈(현재 5문항)
- **`/complete`**: 완료/보상/등록 CTA
- **`/faq`**: FAQ
- **`/api/submit`**: 구글 시트/웹훅 전송(현재는 `GOOGLE_SCRIPT_URL` 있을 때만 외부 전송)

#### 1-2. 데이터/상태
- 콘텐츠: `src/data/contents.ts`
- 퀴즈: `src/types/onboarding.ts`의 `QUIZ_QUESTIONS`
- 로컬 저장: `src/lib/storage.ts` (`localStorage` 기반)
- 제출: `src/lib/api.ts` → `POST /api/submit`

#### 1-3. UI 시스템(현 상태의 “템플릿 느낌” 원인 후보)
- **색상 토큰이 3갈래로 혼재**
  - `globals.css`의 CSS 변수(`--idus-orange`) + `@apply bg-idus-orange …`
  - `tailwind.config.ts`의 `idus.orange`(→ `bg-idus-orange`) + 별도 `idusOrange`
  - 일부 페이지에서 표준 Tailwind(`bg-orange-500`) 직접 사용
- 결과적으로 “페이지마다 채도/명도/여백/그라데이션”이 달라지고, 통일된 브랜드 인상이 약해짐.

### 3) 리뉴얼 목표(제품/콘텐츠 관점)

#### 3-1. 목적 적합성(전환)
- 온보딩의 “끝”은 학습 완료가 아니라 **글로벌 작가 등록(외부 링크 클릭)**.
- 학습은 “모든 정보를 다 제공”이 아니라, **불안/장벽 제거**와 **첫 실행(등록/작품 ON/발송)**을 돕는 “가장 짧은 루트”가 되어야 함.

#### 3-2. 피로도 최소화(인지 부하 관리)
- 한 화면에 **한 메시지/한 행동** 원칙.
- 페이지 당 카드 3개 이하 유지 + 필요 시 콘텐츠를 쪼개서 “다음”으로 넘기기.
- 긴 표/리스트는 **요약 → 상세(접기/외부링크)** 구조.

### 4) 정보구조(IA) 리뉴얼 제안(화면 단위)

#### 4-1. `/` (입력) 리뉴얼: “2단 구성”
- 상단: “왜 지금 시작해야 하는지” 1문장 + 1개 지표(예: 45개국, 배송비 0원)  
- 하단: 입력 폼(필수만) + 개인정보/수집 목적 명확화

권장 변경:
- 입력 단계에서 **폼을 더 짧게**: 이름/연락처/사업자/카테고리만 유지(이미 충족).
- 카테고리는 “선택한 것이 왜 중요한지(금지품목/배송가능)”를 **마이크로 카피**로 안내.

#### 4-2. `/qualification` 리뉴얼: “선택 항목이 안 보이는 문제” 방지
현상: 드래그하면 보이는 것은 **텍스트 대비/배경 토큰 문제**일 가능성이 큼.  
리뉴얼 원칙:
- **텍스트 컬러는 뉴트럴(Gray-900/700)로 고정**, 브랜드 컬러는 배경/칩/CTA에만 제한적으로 사용.
- 카테고리 칩은:
  - 기본: `bg-white border-gray-200 text-gray-900`
  - 선택됨: `bg-idus-orange text-white` + 아이콘(체크/스티커)

#### 4-3. `/learn` 리뉴얼: “지도형 학습”
- 스텝을 카드 3개로 보여주되, 각 카드에
  - “이 스텝을 보면 무엇이 해결되는지(불안 제거)” 1줄
  - “예상 소요” + “완료 보상”은 상단/하단에 분산시키지 않고 **한 영역**에만

#### 4-4. `/learn/step/[id]` 리뉴얼: “스토리텔링 슬라이드”
- 상단 고정: 현재 위치(예: STEP2 / 2 of 3) + 진행바
- 본문: 카드 1개(콘텐츠)
- 하단: 다음 CTA 1개

규칙:
- 섹션 카드 3개 초과 금지(현재는 잘 지켜지고 있음)
- “긴 리스트(판매 가능/불가)”는:
  - 화면: **상위 규칙 3개만**(예: 식품/향수 불가, 캔들 MSDS, 14K 불가)
  - 상세: 링크로 유도([판매 가능/불가능 품목 상세 안내](https://artist-mate.idus.com/b5131e26-6b56-4f1a-81c0-83c9cd66ea9e))

#### 4-5. `/learn/quiz` 리뉴얼: “짧고 확신 주는 확인”
- 현재 개선된 방향(선택 시 체크 표시, 결과 정확 계산)을 유지하되
  - 정답/오답 설명을 더 “다음 행동”으로 연결(예: “이제 등록하러 가기”)

#### 4-6. `/complete` 리뉴얼: “CTA 1개 + 신뢰”
- CTA 버튼은 **1개만 가장 강하게**:  
  `https://artist.idus.com/setting/global-artist/manage`
- “보상 안내”는 과한 그라데이션 대신, **브랜드 오렌지 단색 + 뉴트럴 배경**으로 정돈.

### 5) 디자인 시스템(규칙/가이드)

#### 5-1. 컬러 토큰(권장: 단일 출처)
- **원칙**: Tailwind 토큰을 단일 소스로 만들고, 페이지에서 `orange-500` 같은 임의 색상 사용을 금지.
- 제안 토큰(의미 기반)
  - `brand/primary`: idus orange
  - `brand/primaryHover`
  - `surface/base`: white
  - `surface/raised`: white + shadow
  - `text/primary`: gray-900
  - `text/secondary`: gray-600
  - `border/default`: gray-200
  - `state/success`: green-500
  - `state/danger`: red-500

#### 5-2. 타이포(권장)
##### 폰트: Pretendard Variable 적용(권장)
- **Pretendard**는 한국어 UI에서 가독성과 “제품 같은 완성도”가 높고, 굵기 분화(Variable)로 위계를 정교하게 만들기 좋습니다.  
  레퍼런스: [orioncactus/pretendard](https://github.com/orioncactus/pretendard)

##### 구현 가이드(Next.js)
- 목표: 렌더링 안정성(FOIT/FOUT 최소화) + weight 위계 정교화
- 선택지:
  - **A안(권장)**: `next/font/local`로 `public/fonts/pretendard/PretendardVariable.woff2` 추가 후 사용
  - B안: CDN import(운영/정책에 따라 허용 시)

##### Weight(굵기) 위계(“bold 강도” 세분화)
- 원칙: “700 남발” 금지. 화면 요소의 역할에 따라 굵기를 쪼개면 즉시 고급스러워집니다.
- 권장 스케일(모바일 기준)
  - **H1**: 24px / **800**
  - **H2**: 20px / **700**
  - **H3**: 16px / **650**
  - **Body**: 15px / **500**
  - **Label/Meta**: 12–13px / **500**
  - **Caption/Hint**: 12px / **450**
  - **Disabled/Secondary**: 12–15px / **450** + 색상으로 약화

##### 문장/강조 규칙(가독성)
- 숫자/지표(“45개국/0원/30분”)는 **숫자만 700**, 문장은 500 유지
- 모바일 한 줄 길이 기준: 한 문장 22~32자 내 권장

#### 5-3. 레이아웃/그리드
- Max width: 720px(폼/완료), 896px(학습)
- Spacing: 8pt 시스템(8/16/24/32)
- Radius: 16px 기본, CTA/칩은 9999px(필)

#### 5-4. 컴포넌트 규칙
- `Button`: Primary 1개/화면, Secondary는 “뒤로/나중에”
- `Card`: Elevated는 **CTA/결과 카드**에만 사용, 나머지는 Outlined
- `Tag(Chip)`: 선택 상태만 오렌지 채움
- `Icon`: 원칙적으로 **브랜드 에셋 우선**, 부족하면 `lucide-react` 같은 라인 아이콘 추가 검토

#### 5-5. 모션(절제된 고급스러움)
- 과한 bounce/gradient 금지
- 기본: fade-in(120ms), slide-up(160ms) 정도로 “부드럽게”

### 7) Railway 백엔드 배포 아키텍처(구글 시트/이벤트 로깅)

#### 7-1. 왜 백엔드가 필요한가?
- Vercel의 Next API로도 가능하지만, 운영 관점에서
  - **시크릿(시트 키/서비스 계정) 분리**
  - **레이트리밋/스팸 방지**
  - **재시도/큐(실패 복구)**
  - **이벤트 스키마 버저닝**
을 위해 Railway 백엔드로 분리하는 것이 유리합니다.

#### 7-2. 추천 구조
##### Apps Script 역할 Railway 이관(요청 반영)
목표: **Sheets는 “아카이빙(로그)”만**, 처리 로직은 Railway로 이전해 속도/안정성/확장성을 확보합니다.

- **Front(Vercel)**: 온보딩 UI + 이벤트 발생(가벼운 호출)
- **Backend(Railway)**: 검증/중복 방지/리트라이/집계/속도 최적화
- **Google Sheets**: 최종 아카이빙(append-only 또는 최신 상태 스냅샷)

권장 엔드포인트:
- `POST /v1/events`  
  - 이벤트 타입: `form_submitted`, `qualified`, `step_completed`, `quiz_completed`, `registration_clicked`, `completed`
- `POST /v1/sheets/append` (내부용; 서버에서만 호출 권장)

#### 7-3. 데이터 품질(중복/업데이트)
현재 구조는 “submit을 여러 번 호출”하는 형태라서 시트에 중복 row가 생길 수 있습니다.  
리뉴얼에서는 아래 2개 중 택1:
- **옵션 A(권장)**: 백엔드 DB(Postgres)에서 `phoneNumber`를 키로 upsert 후, 시트는 “일자별 스냅샷” 또는 “최신 상태”만 반영
- 옵션 B: 시트에서 phoneNumber로 row 검색 후 업데이트(구현 난이도↑, Google Sheets API 호출 증가)

**추가 권장(빠른 처리/안정성)**:
- 이벤트 수집 API는 **즉시 200 OK** 응답 후 비동기로 처리(큐/잡)
- Sheets write는 느릴 수 있으므로 **DB → 배치로 Sheets**가 가장 안정적

#### 7-4. 보안/운영
- 레이트리밋: IP/전화번호 기준
- 서명: `X-Signature`(HMAC)로 프론트 요청 위조 방지
- PII(연락처) 마스킹 저장 옵션(예: 시트에는 `010-****-1234` + 내부 DB에 원문)

Railway 운영 참고:
- 로그 확인 가이드: [Railway Logs 가이드](https://docs.railway.com/guides/logs?utm_source=openai)
- 장애 확인: [Railway Status](https://status.railway.com/)

### 11) “장벽 최소화” 기준으로 재점검 & 업스케일(추가 보완)

#### 11-1. 작가님 장벽을 줄이는 5가지 원칙
- **원칙 1 — 첫 화면 3초 룰**: “해외 배송비 0원 / 45개국 / 작가님은 2가지만”을 3초 안에 인지
- **원칙 2 — 불안 먼저 제거**: “정산/환율/통관”은 ‘idus가 대신’이 먼저, 상세는 다음
- **원칙 3 — 행동은 1개씩**: 한 화면 Primary는 1개(결정 피로 최소화)
- **원칙 4 — 요약→상세**: 긴 표/리스트는 요약 3개 + 상세 링크
- **원칙 5 — 완료 후 즉시 다음 행동**: 완료 화면에서 CTA 1개로 끝(글로벌 작가 관리)

#### 11-2. 콘텐츠 논리 재점검(현재 흐름 기준)
- STEP1: “왜/무엇/어떻게(물류/정산)” → **불안 제거**
- STEP2: “등록/언어/가능 품목” → **실행 가능성 확보**
- STEP3: “주문/문의/팁/서류” → **운영 자신감**

#### 11-3. 데이터 기반 개선(운영 루프)
- Railway에서 이벤트를 기준으로 **이탈 지점**을 분석해 화면/카피를 반복 개선
- Sheets는 최종 아카이빙(리포트) 역할로만 유지

### 8) 자체 점검 기준(리뉴얼 전/후 평가 프레임)

#### 8-1. 목적 적합성(전환)
- **CTA 클릭률**: `/complete`에서 “글로벌 작가 관리 페이지” 클릭률
- **중간 이탈률**: `/qualification` → `/learn` 진입률, `/learn` → `/quiz` 진입률

#### 8-2. 피로도/난이도
- 화면당 정보량(문장 수/리스트 길이) 제한 준수 여부
- “작가님이 할 일”이 2개로 인지되는지(사용자 테스트 5명)

#### 8-3. 콘텐츠 정확성/누락
- 링크 정상 동작, 로그인 필요 페이지는 “경로 안내” 포함
- 금지 품목/주의 항목이 최신 자료 기반인지(업데이트 주기 명시)

#### 8-4. UI 품질(전문성)
- 컬러 토큰 단일화(혼재 금지)
- 카드/버튼/타이포 스케일 일관성
- 접근성: 대비, 키보드 포커스, 폼 라벨

#### 8-5. 데이터 품질(시트/아카이빙)
- STEP/QUIZ/REG 이벤트가 “한 작가(전화번호)” 기준으로 정합성 있게 누적되는지
- 중복 row 방지(업데이트 전략 명확)

### 9) 실행 로드맵(제안)

#### Phase A — 디자인 시스템 정리(1~2일)
- 토큰 단일화(색/타입/여백)
- `Button/Card/Input/Chip`를 시스템화

#### Phase B — 핵심 화면 리뉴얼(2~4일)
- `/` / `/qualification` / `/learn` / `/complete` 우선

#### Phase C — 학습/퀴즈 정교화(2~3일)
- 카드 3개 이하 원칙 유지하면서 “요약→상세” 구조 확정
- 퀴즈는 “확신/다음 행동” 중심으로 카피 정리

#### Phase D — Railway 백엔드 분리(2~5일)
- 이벤트 수집 API + 시트/DB 저장
- 운영용 대시보드(선택)
